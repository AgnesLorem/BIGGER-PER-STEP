# MVP-004 — Native DataStore Save System

MVP-004 adds durable, server-authoritative player profiles using Roblox native `DataStoreService` and `UpdateAsync`. Server memory remains the sole gameplay source of truth.

## Objective

- Persist normalized schema-v1 player profiles with `DataStoreService:GetDataStore("BiggerPlayerProfiles_v1", Scope)` and key `Player_<UserId>`.
- Serialize every `UpdateAsync`, including load and release, through one FIFO worker per UserId.
- Protect profiles with 180-second session leases and mutation freezing.
- Autosave every 60 seconds, save on leave, and flush concurrently during a 25-second shutdown window.
- Claim offline growth atomically and exactly once.
- Persist Developer Product receipts before returning `PurchaseGranted`.
- Preserve the current public `SaveService` signatures.

Do not add ProfileStore, another dependency, HTTP access, new remotes, client-authoritative state, or production-place publication.

### Exact Limits

```text
Maximum unlocked Growth Upgrades: 64
Maximum completed portals: 256
Maximum stored processed receipts: 5,000
Maximum durable-plus-pending receipts at runtime: 5,000
Maximum content ID length: 64
Maximum PurchaseId length: 128
Maximum safe integer: 9007199254740991
```

- Stored validation checks `ProcessedPurchaseIds` alone and allows at most 5,000 IDs.
- Runtime admission checks `count(Session.PurchaseHistory) + count(Profiles.PendingPurchaseIds)` and allows at most 5,000 IDs.
- `PendingPurchaseIds` is runtime-only and is never a profile field.
- Content IDs match `[A-Za-z0-9_-]+`; Purchase IDs match `[A-Za-z0-9_.:-]+`.
- `Bigger` is an integer from 1 through the maximum safe integer.
- Destruction, Rebirth, and timestamps are integers from 0 through the maximum safe integer.
- Reject NaN, infinity, unsafe integers, malformed tables, invalid known field types, future schema versions, malformed IDs, and oversized collections.
- An equipped Growth Upgrade must also be unlocked.
- Settings accept only schema-v1 boolean keys.
- Invalid stored known fields fail load without overwriting the record.
- Missing valid fields receive schema-v1 defaults; unknown legacy fields are discarded.
- New profiles and timestamp-less legacy profiles receive zero initial offline reward.

Keep `Bigger`; the separate `Bigger` to `Size` migration is outside MVP-004.

## Public Interface Contract

Preserve exactly:

```luau
function SaveService:LoadProfile(Player: Player): Profile
function SaveService:SaveProfile(Player: Player)
function SaveService:CommitTransaction(
	Player: Player,
	TransactionData: {
		PurchaseId: string,
		Rewards: { [any]: any },
	}
): boolean
```

Add only:

```luau
SaveService:IsProfileWritable(Player: Player): boolean
SaveService:FreezeMutations(Player: Player, Reason: string): boolean
SaveService:ReleaseProfile(Player: Player): boolean
```

- `LoadProfile` raises a controlled error after retries; `SessionService` invokes it through `pcall`.
- A failed read never creates a default gameplay session.
- `SaveProfile` waits for its queued write and preserves its no-return signature.
- `CommitTransaction` validates but never redispatches or reapplies `Rewards`; it returns true only after the PurchaseId is durable.
- Freeze and release are idempotent.

## Runtime, Lease, and FIFO Contract

`RuntimeRegistry.Profiles` is keyed by UserId. Each entry owns `Phase`, `OwnerToken`, `LeaseExpiresAt`, `EntitlementProbe`, `WriteQueue`, `Worker`, `PendingPurchaseIds`, `Released`, and `LastError`.

Phases are `Loading`, `Active`, `Releasing`, and `Released`.

1. Create the temporary entry and worker before load.
2. Queue load/acquire as the first worker request.
3. Route every `UpdateAsync` through that worker.
4. Allow different UserIds to write concurrently, but never overlap writes for one UserId.
5. Coalesce duplicate autosaves only; never discard receipt or final requests.
6. Capture the session snapshot and exact pending IDs immediately before the request's first DataStore call.
7. Use callbacks/coroutine completion, not Signal, BindableEvent, or a dependency.
8. Use a 180-second lease.
9. Retry foreign locks after `1, 2, 4, 8` seconds and writes after `1, 2, 4` seconds.
10. Never steal an unexpired foreign lease.
11. Freeze and remove the player on wrong-owner save or release.
12. Clear the lease atomically and terminate the worker only after successful release.
13. Keep a failed final release frozen in `Releasing`; reject a same-UserId rejoin while it remains unreleased.

## Mutation Freeze

Set `Session.ProfileMutationsFrozen = true` before PlayerRemoving, shutdown, final release, or ownership loss. It never returns to false during that session.

Fail closed without changing persistent fields in `GrowthService:SetBigger`, Destruction reward mutation, `RewardService:Grant`, new receipt dispatch, stored entitlement mutation, and every other direct profile mutation found during the audit.

## Offline Progression

During atomic lock acquisition:

```text
OfflineSeconds = clamp(Now - max(LastActiveTimestamp, LastLogoutTimestamp), 0, 28800)
OfflineReward = floor(OfflineSeconds / 60 × 30 × GrowthMultiplier)
```

- New and timestamp-less legacy profiles receive zero.
- Apply entitlement reconciliation, reward, lease, `LastActiveTimestamp = Now`, and `LastSaveTimestamp = Now` in one transform.
- Normal writes do not change `LastLogoutTimestamp`.
- Final release sets active, logout, and save timestamps to `Now` and clears the lease.
- Reject overflow and never claim the same interval twice.

## Durable Receipt Contract

`Session.PurchaseHistory` contains only durable Purchase IDs. `Profiles.PendingPurchaseIds` contains runtime-only IDs already dispatched but not confirmed durable.

- Durable duplicates return `PurchaseGranted`.
- Pending duplicates retry persistence without redispatch.
- Mark new IDs pending before dispatch and remove them only when dispatch itself fails.
- Persist the complete authoritative snapshot plus the request's exact included pending IDs.
- Promote only IDs included in a successful request and resolve all included waiters.
- Return `PurchaseGranted` only after durability; failed or ambiguous writes return `NotProcessedYet`.
- Never roll back an in-memory reward solely because a write response is ambiguous.
- Never prune, expire, evict, or reuse receipt IDs.
- At combined capacity 5,000, existing duplicates still succeed; new receipts are not dispatched, return `NotProcessedYet`, and emit a Critical operational log.
