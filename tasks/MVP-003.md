# MVP-003 — Whole-Project Quality, Security & Bug Gate

MVP-003 permanently owns project-wide defect auditing, security hardening, runtime resilience, validation, and release verification. It is a standing release gate, not a one-time patch milestone.

---

## Permanent Policy

- Process findings strictly: **Critical → High → Medium → Low**.
- Do not begin a lower severity while a confirmed higher-severity finding remains open unless the higher item is recorded as blocked with evidence.
- Fix defects in their actual source files. Do not move gameplay or Core code into an MVP-003-specific folder.
- Local files under `src/` remain the Git source of truth.
- Never create `MVP-X.5` milestones for bug fixes; add an MVP-003 audit cycle instead.
- A later MVP cannot be release-ready while confirmed Critical or High findings remain open.
- Preserve user changes. Never use reset, checkout, clean, force-push, or bulk overwrite.
- Do not push implementation without explicit user review and approval.
- If a defect requires a file outside the exact cycle scope, record the path and reason here, extend the exact scope, and stop for Tech Lead approval before editing it.
- The Tech Lead conditionally approved execution on 2026-07-15. After the final execution amendments are committed, no further architecture review is required unless an actual Roblox Engine limitation or an out-of-scope source defect is discovered.

### Severity Definitions

1. **Critical**: exploitable authoritative-state mutation, data loss/corruption, or a boot/runtime failure blocking the core loop.
2. **High**: broken progression, invalid world/session lifecycle, reward duplication, ownership isolation failure, or a player-trapping failure.
3. **Medium**: recoverability, cleanup, resilience, or architecture-boundary defects that can degrade a live server.
4. **Low**: lint, formatting, dead code, naming drift, duplicated definitions, or maintainability defects without immediate gameplay impact.

### Required Initial-Cycle Acceptance Contracts

- `SessionService:RecoverToLobby(Player, Reason)` is idempotent, ignores the current source state, restores `InLobby`, clears `CurrentWorldInstanceId`, and clears objective presentation. Entry, completion, teleport, and cleanup failures must still complete runtime cleanup, objective cleanup, portal debounce cleanup, and lobby recovery if an earlier cleanup operation fails. Test unexpected `EnteringWorld`, `InWorld`, and `Returning` states, including repeated recovery.
- Duplicate world allocation must preserve a valid active world when registry ownership and `CurrentWorldInstanceId` consistently describe it. Reject that duplicate without recovery; call `RecoverPlayer` only for stale or inconsistent ownership.
- `RecoverPlayer` isolates disconnect, destroy, registry removal, and debounce cleanup; then attempts `SessionService:RecoverToLobby` through protected execution; only afterward may it call `Player:LoadCharacter`. No earlier recovery step may throw past the authoritative lobby-state attempt.
- `StompValidation` rejects NaN or infinity in every numeric argument, non-positive hitbox half sizes, negative padding, and non-positive maximum height. Unit coverage includes NaN position, infinite position, NaN velocity, invalid dimensions, excessive height, side contact, and a valid stomp.
- Movement growth uses server-owned `MovementConfig.MaximumRewardSpeed` and `MovementConfig.DistanceSlack`, never `Humanoid.WalkSpeed` as its primary security boundary, and samples intervals with `time()` rather than `tick()`. Unit coverage includes normal motion, scheduler jitter, teleport displacement, zero/negative delta, NaN, and infinity.
- `ReturnSpawn` is preflighted before entry and resolved again immediately before return. If it disappears while the player is inside WORLD1, reward remains exactly once, cleanup completes, the player is not left in `Returning` or a destroyed world, and recovery uses no arbitrary CFrame.
- `stylua.toml` uses `line_endings = "Unix"`; `.gitattributes` applies LF to `*.lua`, `*.luau`, `*.toml`, `*.json`, and `*.md`. Do not bulk-format unrelated files.
- Before deleting GrowthSource entries, GrowthFormula helpers, or `GrowthTypes`, search all of `src` and `tests`; never delete a current caller's symbol. Any required out-of-scope edit triggers scope extension and a Tech Lead stop before editing.
- Raw Git baseline evidence is stored in the initial dated audit report, not inferred from prose in this file.
- The Studio recovery test is a temporary `ServerStorage.MVP003Tests.SessionRecovery` ModuleScript mirrored from `tests/studio/session_recovery.luau`, parity-checked, executed only in Play Solo with a real Player, and removed after verification. Production runtime never requires or registers it.

## Offline Release Gate

Every audit cycle must run, in order:

1. Focused unit tests for changed validation logic.
2. Scoped StyLua on every changed Luau file.
3. Selene on every changed Luau file, then the full `src` tree before release.
4. `git diff --check`.
5. `rojo build default.project.json` to a temporary path.

All commands must exit 0 before Studio verification begins.

---

## Studio Release Gate

Because Rojo live sync is currently unreliable:

1. Write final source locally.
2. Mirror identical source into the original Studio place using MCP.
3. Compare normalized Studio Source with local source.
4. Stop if parity differs.
5. Never open a generated place.
6. Never create fake Player objects.
7. Run Play Solo twice for gameplay/state changes.
8. Record Studio Output for each run.
9. Security scans must include legacy `Message` and `Hint` instances plus CollectionService tags; scanning only TextLabel, TextButton, TextBox, source, values, and attributes is insufficient.
10. Studio-only remediation is not complete until the original cloud place is published and a new Play session proves the removed content does not return.

Manual tests must be labeled manual. Never claim they were MCP-verified.

---

## Multiplayer Release Gate

Run multiplayer verification whenever ownership, replication, networking, or private instances are involved. Verify:

- separate player-owned runtime worlds;
- no cross-player objective completion;
- rewards update only the authoritative owner session;
- PlayerRemoving cleanup;
- repeated portal contacts do not create duplicates.

If MCP cannot access the local server DataModel, the user may execute the two-player test manually and provide the evidence.
