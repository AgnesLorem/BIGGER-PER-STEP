## Requirements

### 1. Git Safety
- Run `git status --short`.
- Stop if the working tree is not clean.
- Verify the current branch is `main`.
- Run `git pull --ff-only origin main`.
- Re-run `git status --short`.

### 2. Portal Entry
- Portal requirement is `Derived Level >= 5`.
- PortalService reads server-memory session state and derives level through current progression logic.
- PortalService rejects entry unless the session is `InLobby`.
- Per-player portal debounce prevents duplicate runtime worlds.

### 3. Private World Runtime
- WORLD1 template is isolated under `ServerStorage.WorldTemplates`.
- Runtime clones live under `Workspace.RuntimeWorlds`.
- Runtime clones store `OwnerUserId` and a unique `WorldInstanceId`.
- Runtime worlds are spatially offset so multiple players do not overlap.
- Failures during creation, marker resolution, initialization, or teleport restore the player to `InLobby`.

### 4. Objective Completion
- Spider objective uses `Spider.ObjectiveHitbox`.
- Stomp is valid only for the owning player, while `InWorld`, with objective active and incomplete, with HumanoidRootPart above the hitbox and moving downward.
- DestructionService locks objective completion before granting rewards.
- Completion grants exactly `+1 Destruction`, updates presentation mirrors, runs replaceable placeholder effect, returns the player, cleans the runtime world, and restores `InLobby`.

### 5. Objective UI
- Objective UI uses server-controlled Player Attributes only:
  - `CurrentWorldId`
  - `ObjectiveActive`
  - `ObjectiveText`
  - `ObjectiveReward`
- Client renders a bottom-right panel while `ObjectiveActive` is true.
